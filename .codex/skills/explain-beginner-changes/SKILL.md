# 初心者向けに変更内容を説明する

## 目的

このスキルは、実装したコードを初心者が理解できるように説明するためのものです。

ただ「このファイルを変更しました」と言うだけではなく、次のことが分かるようにしてください。

- 何を作ったのか
- ブラウザ操作からデータ保存まで、処理がどう流れるのか
- 各ファイルがどんな役割を持つのか
- なぜその構成にしたのか
- 初心者がどこを意識すればよいのか

## 進め方

1. 変更されたファイルを確認する
   - `git diff` やファイルパスを見て、変更内容を把握します。
   - 役割ごとに整理します。
   - 例: route、request、controller、service、repository、model、migration、seeder、blade、css、js、test。

2. 処理の流れを説明する
   - Web アプリでは、ブラウザ操作から順番に説明します。
   - 例: フォーム送信 → Route → Controller → Service → Repository → Model / DB → Blade。
   - 実際の機能名を使って説明します。

3. ファイルごとの役割を説明する
   - そのファイルが担当することを書きます。
   - そのファイルに書きすぎない方がよいことも書きます。
   - 初心者が混同しやすい点を補足します。

4. 実装判断を説明する
   - なぜその書き方にしたのかを書きます。
   - もっと簡単な方法、もっと複雑な方法がある場合は、なぜ今回は採用しなかったかを書きます。
   - 覚えてほしい考え方を明確にします。

5. 自分で確認する方法を書く
   - ブラウザで何を確認すればよいかを書きます。
   - 必要なら `php artisan test` などのコマンドも書きます。

## 出力形式

特に指定がなければ、次の形で出力してください。

```markdown
## 何を作ったか
<機能の概要>

## 処理の流れ
1. <ブラウザやフォームでの操作>
2. <Route / Controller の処理>
3. <Service / Repository / Model / DB の処理>
4. <画面に戻る流れ>

## ファイルごとの役割
- <ファイルパス>: <何をするファイルか / 何に注目するか>

## 意識するポイント
- <概念>: <なぜ大事か>

## 初心者がつまずきやすいところ
- <つまずきポイント>: <考え方>

## 自分で確認すること
- <コマンドまたはブラウザでの確認>
```

## 書き方のルール

- 日本語で書きます。
- 短い段落で、具体例を使います。
- 難しいところを隠さず、順番に分解して説明します。
- すべての構文を細かく説明しすぎないでください。
- 「このファイルの責任」と「データの流れ」を重視してください。
- テストや確認コマンドも、学習の一部として説明してください。
