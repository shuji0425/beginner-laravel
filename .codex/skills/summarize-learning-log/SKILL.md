# 学習ログをまとめる

## 目的

このスキルは、会話の内容をそのまま貼り付けるのではなく、初心者があとから読み返して理解できる「学習ログ」に整理するためのものです。

育成者の立場で、次のことが分かるようにまとめてください。

- 何を作ろうとしていたのか
- どんな依頼があったのか
- それに対して何を実装・修正したのか
- なぜその判断をしたのか
- 初心者が覚えておくべき考え方は何か

## 進め方

1. 学習の目的を確認する
   - 最初の依頼で何を目指していたかをまとめます。
   - 途中で方針が変わった場合は、その変化も記録します。
   - `Todo CRUD`、`Seeder`、`FormRequest`、`setup.sh` など、実際に出てきた言葉を使います。

2. 会話を作業単位に分ける
   - 3〜8個くらいの区切りにします。
   - 各区切りに「依頼」「対応」「学習ポイント」を入れます。
   - 実装した順番が分かるようにします。

3. 作成・変更したファイルを整理する
   - 分かる範囲でファイルパスを書きます。
   - 新しく作ったファイルと、変更したファイルを区別します。
   - 実行した確認コマンドも書きます。

4. 判断理由を書く
   - なぜ初心者向けにその構成にしたのかを書きます。
   - あえて使わなかったものも書きます。
   - 例: Docker、MySQL、Vite、複雑な interface など。

5. 最後に現在地を書く
   - 今どこまで動くのかを書きます。
   - 次に何を試すとよいかを書きます。
   - 未解決のことがあれば、必要なものだけ書きます。

## 出力形式

特に指定がなければ、次の形で出力してください。

```markdown
# 学習ログ: <テーマ>

## 目的
<この作業で目指したこと>

## 会話と作業の流れ
1. <依頼 / 対応 / 学習ポイント>
2. <依頼 / 対応 / 学習ポイント>

## 作成・変更したもの
- <ファイルパス>: <役割>

## 学んでほしいポイント
- <概念>: <初心者向けの説明>

## 確認したこと
- <コマンドまたは確認内容>: <結果>

## 現在の状態
<今できること>

## 次に進むなら
- <小さな次の一歩>
```

## 書き方のルール

- 日本語で書きます。
- 会話をそのまま長く貼り付けないでください。
- 「対応しました」だけで終わらせず、何をして、なぜそうしたかを書いてください。
- 初心者向けにやさしく書きますが、技術的にあいまいな説明にはしないでください。
- 抽象的な説明より、具体的なファイル名や Laravel の用語を使ってください。
- ログ保存を依頼された場合は、指定された場所に Markdown ファイルとして作成・更新してください。
- 保存先の指定がない場合は、プロジェクト内の `logs/YYYY-MM-DD-learning-log.md` を優先してください。
- `logs/` は個人用の履歴置き場です。Git のコミット対象にしない前提で扱ってください。
- 履歴を作成するときは、必要に応じて `logs/` ディレクトリを作成してください。
